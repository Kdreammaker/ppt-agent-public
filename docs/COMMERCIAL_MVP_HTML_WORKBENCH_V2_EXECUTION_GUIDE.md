# Commercial MVP HTML Workbench V2 Execution Guide

## Purpose

This plan guides the next implementation sequence after the first HTML
workbench vertical slice. It is separate from the PRD: the PRD defines product
requirements, while this plan defines the work order, acceptance evidence, and
known traps for the next worker.

The next work must not polish the current static demo deck as if it were the
product. The current `web/commercial-mvp-html-workbench/workbench-data.json`
deck is a public-safe fixture only. It is not copied from the sample files, and
it was not generated by Python, but it is still hardcoded demo data and must be
treated as such.

## Goal

Build HTML Workbench V2 foundation:

- real direct text editing that works for a user on the rendered canvas.
- persistent editor toolbar.
- automatic fixed-canvas fit and zoom behavior.
- rich text runs for partial styling inside one text object.
- text style roles for Title, H1, H2, H3, Body, Caption, and Bullet.
- master style, theme token, layout recipe, and component recipe references.
- multi-select, align/distribute, box-based resize, shape/image fixed-step
  rotation, shape/image flip, and shape corner radius controls.
- safe server/asset-system design package path.
- AI revision memory distinct from undo/redo.
- account/project Style Memory distinct from revision memory.
- Reference Design Library scaffolding for local-host-AI extracted, design-only
  recipes from existing PPTX/PDF/HTML decks.
- published HTML canvas viewer scaffolding for read-only link sharing.
- referral scaffolding and the initial Free plus one Paid entitlement posture.
- dark/light app chrome and English/Korean UI switching.
- honest export hooks preserved.

## Non-Negotiable Boundaries

- Do not copy or hardcode `benchmark HTML reference` or `benchmark PPTX reference`.
- Do not copy the benchmark brand content, exact coordinates, raw DOM, image URLs, filenames,
  or sample-specific business text.
- Do not use Python to directly fabricate a final sample-like deck.
- Do not present the current static fixture as host-AI generated output.
- Do not claim PDF/PPTX export completion without a real host-AI result
  reference.
- Do not add a third mode beyond Assistant and Auto.
- Do not build embedded backend AI chat.
- Do not expose raw prompts, raw sources, local paths, filenames, external document identifiers,
  package internals, credentials, DB URLs, encoded asset assets, private image URLs, or
  backend chain-of-thought in public summaries or handoff envelopes.
- Do not imply production Supabase, Cloudflare, or Vercel deployment unless the
  work actually performs an explicitly approved remote deployment or mutation.

## Required Architecture

### Workbench State

Introduce a workbench state shape that separates:

- `design_package`: safe package id, version, manifest hash, source kind, and
  entitlement/source status.
- `theme_tokens`: palette, typography roles, spacing, radius, stroke,
  elevation, and app/canvas chrome tokens.
- `master_styles`: shared typography role defaults, palette defaults, shape
  defaults, and common slide chrome. V2 master style is not a full slide layout
  master.
- `text_style_roles`: required roles `Title`, `H1`, `H2`, `H3`, `Body`,
  `Caption`, and `Bullet`, each with font family token, font size, weight,
  color, line height, paragraph spacing, and overflow policy. Letter spacing
  defaults to `0`.
- `layout_recipes`: editable regions, hierarchy, density, anchors, and asset
  slots.
- `component_recipes`: reusable cards, timelines, tables, metric blocks,
  diagrams, image-plus-insight blocks, icon grids, and closing blocks.
- `slides`: content objects that reference master/layout/component ids.
- `objects`: editable instances with geometry, z-order, style, safe refs, and
  optional overrides.
- `paragraphs`: text object paragraphs with role, alignment, bullet flag,
  bullet level, spacing, and overflow policy.
- `text_runs`: rich text ranges inside paragraphs.
- `object_transforms`: shape/image rotation, flipX, flipY, box-based resize
  bounds, and shape corner radius or radius token.
- `revision_memory`: public-safe prior AI work, handoffs, proposals,
  accept/reject decisions, and local edit summaries.
- `style_memory`: account/project preference profile summaries, including
  accepted/rejected recipes, preferred text-role defaults, palette decisions,
  density preferences, and recurring local edit patterns. This must remain
  user-visible and deletable.
- `reference_design_recipes`: content-free design recipes extracted by local
  host AI from existing PPTX/PDF/HTML decks, with source-content and
  original-file storage blocked.
- `published_views`: read-only published deck-version metadata for viewer links,
  watermark/attribution state, and future expiration/password/noindex flags.
- `referral_entitlement`: safe free-credit/referral attribution status where
  relevant to the account UI.
- `fixture_metadata`: explicitly marks demo fixtures.

### Design Package Source

Support these source kinds:

- `tracked_doc`: local baseline guide.
- `server_manifest`: product server supplied guide/theme/master-style/layout
  package.
- `approved_asset_system`: asset-system supplied layouts, fonts, palettes,
  icons, illustrations, or style tokens.
- `imported_approved_asset_result`: already-approved asset-system output
  internalized into PPT Maker's own design-package/library model.

Asset-system design package use requires:

- entitlement check.
- approved package status.
- materialization/evidence where relevant.
- safe ids only in public UI/report/handoff.
- Starter/Manager block for Leader-only details.
- no package internals in public state.
- no raw reference holdings, assetization workflow internals, approval logs, or
  external workspace paths imported into PPT Maker.

### Editor Surface

The primary UI must expose:

- left 10+ slide navigation.
- dominant 16:9 canvas.
- persistent toolbar for text, object, layer, undo/redo, zoom/fit, and insert
  actions.
- multi-select controls for align left/center/right, align top/middle/bottom,
  distribute horizontally/vertically, align-to-selection, and align-to-slide.
- shape/image controls for rotate -15, rotate +15, rotate -90, rotate +90,
  reset rotation, flip horizontal, flip vertical, and corner radius where
  applicable.
- right inspector only for secondary precision edits.
- Assistant/Auto explanation as host-AI behavior, visually separate from slide
  edit controls.
- hidden diagnostics in normal mode.
- dark/light app chrome.
- English/Korean locale switch.

### Text Editing

Acceptance requires real user-level editing:

- click or double-click visible text object.
- caret appears at the intended canvas text.
- user can type Korean text.
- selected text range can receive partial bold/color/font/font-size changes.
- text object uses one of `Title`, `H1`, `H2`, `H3`, `Body`, `Caption`, or
  `Bullet`.
- bullet lists are stored as paragraph-level bullet metadata with bounded
  levels, not only visible bullet characters.
- commit mutates workbench state.
- undo/redo restores local edit states.
- AI revision memory remains unchanged by undo/redo unless an explicit revision
  action occurs.
- text box rotation is not implemented in V2.

### Object Transform Editing

V2 object transforms must keep implementation bounded:

- resizing uses the object's stored box, even when the object is rotated.
- align and distribute operate on stored boxes.
- rotation applies only to PPT Maker-created shapes and inserted images.
- rotation uses fixed buttons only: counterclockwise/clockwise 15 degrees,
  counterclockwise/clockwise 90 degrees, and reset.
- freeform rotation is deferred.
- horizontal and vertical flip apply only to shapes and inserted images.
- text boxes must show rotation controls as unavailable or omit them entirely.
- corner radius applies to shapes through a numeric field or safe radius token.

### Canvas Fit

The canvas remains a fixed logical 16:9 surface, but display scale must be
automatic by default:

- measure stage area with side panels included.
- fit within available width and height.
- support explicit fit/zoom controls.
- keep geometry stable; display scale must not rewrite object coordinates.
- work at 1440, 1600, 1920, ultrawide, narrow width, and browser text scaling.

### Reference Design Library

The next product path should support a library for existing slide styles, but
the extraction boundary is local-host-AI-first:

- local host AI or local workspace tools analyze PPTX/PDF/HTML decks.
- PPT Maker receives only a design-only recipe payload.
- original files, source slide screenshots, source text, raw prompts, local
  paths, and private URLs are not stored on the product server.
- stored recipes may include theme token-like values, text-role defaults, layout/component
  recipes, slide archetypes, relative geometry, and content-free previews.
- the library must support reuse in future host-AI generation or revision
  handoffs through safe recipe ids.

### Style Memory

Style Memory is not undo/redo and not hidden model memory. It is a user-visible
account/project preference profile stored in Supabase when backend work is in
scope.

Useful signals include:

- repeated user changes to text-role size, weight, font, or color.
- accepted/rejected layout and component recipes.
- preferred palette families and blocked palette families.
- density, margin, logo placement, and image/text balance preferences.
- project-specific notes such as Korean font preference or dark-background
  avoidance.

The user must be able to inspect, reset, or delete Style Memory. Public
handoffs may include only compact, content-free preference summaries.

### Published Viewer

Published HTML canvas viewer links are the primary sharing primitive. They
should publish immutable deck versions and render them read-only. Viewer links
are not collaborative editors and must not expose the editing API, raw
workbench state, private asset URLs, local paths, package internals, or
credentials.

### Initial Entitlement Model

Start with Free plus one Paid plan:

- Free: limited credits, referral-earned credits, editor value preview,
  watermark/attribution on viewer links, and limited design-memory/library
  capacity.
- Paid: practical editor access, no visible per-edit credit accounting for
  normal workflows, Style Memory, useful Reference Design Library capacity,
  watermark-free viewer sharing, approved design packages where entitled, and
  internal fair-use controls for abuse.

Two paid plans, team workspace, native collaboration, and broad cloud-drive
file management are deferred.

1. **Right Inspector Product IA**
   - Move `Master style`, `AI revision memory`, `Reference Design Library`,
     `Style Memory`, `Published viewer`, `Referral and credits`, `Local
     font/image links`, and `Host-AI export` out of the always-visible right
     inspector.
   - Keep the inspector focused on selected-object controls and secondary
     precision edits.

     affordances when still needed for QA.

   - Replace raw `master_style_id`/role/default displays with a deck/theme

     semantics.
   - The selected-object inspector may show the resolved text role/token, but
     deck-wide theme management must be separate.

   - Recipes should capture palette, typography roles, layout archetypes,
     component recipes, image-slot treatment, chart/table style, spacing
     rhythm, density, and synthetic placeholder previews.

   - Replace safe-reference-only posture with actual approved package/design
     package consumption when entitlement and manifest evidence exist.
   - Record approved package refs, license/version/checksum evidence, safe ids,
     slot mapping, render-use evidence, and boundary validation.
   - Continue to say "asset-system ready" rather than "asset-system fused"
     until actual approved assets are rendered.

5. **Generated Work State Path**
   - Keep `workbench-data.json` fixture-only.
   - Add or harden a host-AI/server-generated work-state loader so the editor
     can open non-fixture decks with design package ids, object records,
     revision memory, export status, and safe asset refs.

   - Use the benchmark family for scoring, not copying.
   - Add visual QA signals for image/media density, shape/component richness,
     slide-to-slide variation, text overflow, collision, hierarchy, chart/table
     quality, and first-slide commercial impact.
   - The next demo/generation path should close the current gap where the
     fixture has far fewer image slots, shapes, and visual components than the
